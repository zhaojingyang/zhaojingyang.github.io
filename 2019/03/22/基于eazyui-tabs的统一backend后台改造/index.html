基于eazyui tabs的统一backend后台改造

发表于 2019-03-22 | 分类于 有点乱 | | 阅读次数: |
字数统计: 1.1k 字 | 阅读时长 ≈ 4 分钟

场景

  • 公司内有若干个项目,对于一个运营人员需要在同一个系统内操作各系统,提高其工作效率，避免在浏览器内切换页面。

  • 公司初期可能是一个一个war包打天下将若干个项目放到同一个war包项目内，随后续发展将war包内某些模块拆分至新的系统。

  • 原web系统使用的eazyui，index页面左边是菜单，右边是内容展示区域，当点击菜单后右面使用eazyui tabs新增一个tab，tab的content为待展示页面的内容，但是会共用backend web项目的静态资源[js css img]等。

  • 对于之前大家在同一个war项目开发可以在本地直接调试，但是对于迁移出去的项目，页面项目只是待显示的html内容(不包含 等标签)，样式等是阉割掉的，要么将页面配到开发环境上调试，要么自己本地copy一份web项目样式;另由于新页面将内容加到tabs的content内，理论上就是加到同一个html页面内，当遇到打开的页面有相同的id等就会出现各种问题。

  • Eazyui tabs见 easyui 文档

  • 原项目原理

    • 点击页面增加tab页

解决方案

  • 在eazyui tabs内嵌一个iframe,iframe的src为待显示的页面

  • 整个项目不能一起迁移所以需要增加兼容方式，在原菜单中增加一个打开方式[div:iframe]，addTab方法增加兼容根据打开方式调用不同的方法，index页面刷新菜单调用新的js方法。

  • 各项目组自己管理自己的静态资源,将原页面补全

  • 原tab页内需要关闭tab页，现在是跨frame了可以使用html5 postMessage解决

后记

​ 这样各项目管理自己的静态资源，各页面也都是完整的html页面，本地调试也都更方便了。backend也不会和各项目耦合在一起，职责更单一了，也不会因不同页面有相同的id而引发各种问题。

​ 关于统一管理后台需要有统一的sso支持，现公司使用的是基于url上都加一个标志符来解决的，需要大家将各自的url都加上登录后的标志key，不是很友好。

每个项目组的人都说这个backend很垃圾，原项目负责人离职交接给我们组，我们组相应你们的需求给优化，但是让你们第一步迁移个静态资源就各种推脱不想做。最后还是我们项目组人切入你们项目帮你们改。真不知道说什么好。
