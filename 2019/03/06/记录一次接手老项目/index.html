记录一次接手老项目

发表于 2019-03-06 | 分类于 有点乱 | | 阅读次数: |
字数统计: 1.3k 字 | 阅读时长 ≈ 6 分钟

​ 接手一个项目,项目原开发人员已离职,只有一个文档上面写了代码的git地址还有各环境的数据库连接.经过一下午的排查总算将项目启动了。记录下排查问题的步骤。

  • 拉取项目并导入idea

  • 打包项目没有报错,启动项目提示日志有问题,经排查项目使用的是log4j配置文件用了${catalina.home}，gg后发现 log4j catalina.home配置，先改成本地目录解决。

    修改成如下方式页面加载正常

  • 再次启动项目登录后还是放回登录页面

    • 怀疑是有鉴权的filter,查看web.xml发现有spring-session相关内容，排查redis链接等正常

    • debug登录发现登录成功并返回,但是页面有个redirect到index,但是index的action断点未进入

    • 继续排查在interceptor包下类打断点,发现登录成功后进入UserLoginInterceptor，但是request.getSession().getAttribute(“sysUser”)为null导致重新返回登录页

    • 大概定位到问题点,继续debug登录和上面的interceptor发现两次获取的sessionId不一致，说明两次访问不是同一个session,尝试debug request.getSession至SessionRepositoryFilter.public HttpSessionWrapper getSession(boolean create)查看代码有些复杂没有搞定。

    • 登录到redis看到每次登录是有新的session写入且key是相同的,说明写入没问题，还是再次访问获取的地方有问题。

    • 排查配置文件发下面配置有domainName，改成localhost重新发布成功，

      进入此类发现writeCookieValue中使用domainName如下，到此问题解决

    • 查看文档java-custom-cookie

      • domainName - allows specifying a specific domain name to be used for the cookie. This option is simple to understand, but will likely require a different configuration between development and production environments. See domainNamePattern as an alternative.
      • domainNamePattern - a case insensitive pattern used to extract the domain name from the HttpServletRequest#getServerName(). The pattern should provide a single grouping used to extract the value of the cookie domain. If the regular expression does not match, no domain is set and the existing domain will be used. If the regular expression matches, the first grouping will be used as the domain.

    • 暂时问题得以解决，后续会将此配置切到各环境的配置文件中。

  • 后记

    ​ 自己对spring-session不是很了解，本次排查走了很多弯路。原项目也是很烂，配置有些在配置文件中有些还没有，对于此问题，我相信原开发在自己调试时候也会去调整domainName，对开发不友好，再离职交接最好能有点责任心，将哪些需要注意的地方说明下吧。

    ​ 记得有项研究表明，能给你工作提供帮助的大概率不是你很亲近或很要好的人，往往是一面”君子之交”，可能在某个公司给人留下了好的印象，然后有工作机会时候可能会内推你。对于这种项目交接且不回信息的人，一路黑粉。
